::Призначення README::
Яку головну роль відіграє README.md у репозиторії GitHub згідно з уроком? {
=Це вітрина проєкту: опис, інструкції, приклади та презентація ідей автора
~Це технічний файл, який замінює папку .git локально
~Це прихований конфіг для SSH-ключів
#### README — перше, що бачить відвідувач; саме тут формується перше враження про проєкт.
}

::Markdown у README::
У якому форматі зазвичай пишуть README на GitHub? {
=У форматі Markdown з можливістю заголовків, списків, посилань і зображень
~У бінарному форматі PDF, який GitHub конвертує автоматично
~Лише у вигляді HTML без підтримки простого тексту
#### Markdown дає читабельний синтаксис і водночас рендериться в охайний веб-вигляд.
}

::Ліцензія та копіювання::
Що обмежує можливість вільно копіювати й використовувати чужий код з GitHub? {
=Ліцензія репозиторію, яка визначає дозволені способи використання
~Кількість зірочок на репозиторії
~Наявність акаунту GitHub у автора
#### Публічний доступ не означає безумовну свободу: ліцензія задає правила для копіювання та модифікації.
}

::Fork як спосіб вивчення README::
Як учень може детально вивчити оформлення чужого README, не маючи прав редагування в оригіналі? {
=Зробити fork і переглянути файл у режимах Preview та Edit у власній копії
~Надіслати автору пароль від свого акаунту
~Видалити оригінальний репозиторій і створити новий
#### Fork дає повну копію, де можна безпечно досліджувати структуру та синтаксис README.
}

::Preview vs Edit::
Чим корисні режими Preview і Edit при роботі з README? {
=Preview показує готовий вигляд сторінки, Edit — сирий Markdown і прийоми форматування
~Preview видаляє всі зображення, Edit додає їх автоматично
~Preview доступний лише власнику, Edit — усім користувачам інтернету
#### Поєднання режимів допомагає зрозуміти, як синтаксис перетворюється на візуальний результат.
}

::Структурований портфоліо-підхід::
Який підхід до README у портфоліо сподобався автору в прикладі профілю розробника? {
=Блоки з картинкою, посиланням і коротким описом для кожного проєкту
~Один абзац тексту без жодних посилань чи візуалів
~Автоматичний список усіх комітів за останній тиждень
#### Структуровані картки проєктів полегшують роботодавцю пошук потрібної інформації.
}

::Шпаргалка Markdown::
Навіщо автор згадує шпаргалку з Markdown у матеріалах курсу? {
=Щоб мати під рукою інструменти форматування під час оформлення README
~Щоб замінити необхідність вивчати Git
~Щоб автоматично генерувати ліцензію проєкту
#### Шпаргалка прискорює роботу з заголовками, списками, посиланнями та іншими елементами Markdown.
}

::GitHub як колаборація::
Як автор описує суть GitHub у контексті README та відкритих репозиторіїв? {
=Це платформа колаборації, обміну ідеями та взаємного навчання через публічний код
~Це закритий сховищний сервіс лише для резервних копій
~Це інструмент лише для автоматичного деплою без спілкування
#### Відкриті репозиторії стимулюють діалог, копіювання ідей і взаємне вдосконалення.
}

::Пошук користувача::
Як у відео знаходять профіль користувача на GitHub? {
=Через пошук за іменем користувача або ім'ям у репозиторіях
~Лише через платну підписку Copilot
~Через випадковий перегляд trending repositories
#### Пошук за username — базовий спосіб знайти людину та її публічні напрацювання.
}

::Відкриті репозиторії як навчальний ресурс::
Що автор підкреслює щодо відкритих проєктів на GitHub для новачків? {
=Ніщо не заважає дивитися, як інші оформлюють README та впроваджують ідеї
~Відкритий код заборонено читати без письмового дозволу автора
~README відкритих проєктів приховано від неавторизованих користувачів
#### Публічність GitHub перетворює чужі репозиторії на живі підручники з оформлення та структури.
}
