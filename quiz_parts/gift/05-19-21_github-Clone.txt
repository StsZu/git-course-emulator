::Clone — класичний сценарій::
Коли клонування через git clone є типовим першим кроком у реальному проекті? {
=Коли вас додають до вже активного репозиторію, де інші вже комітять зміни
~Коли потрібно лише разово скачати zip без історії
~Коли локальний .git уже є, але порожній
#### У відео це сценарій «приєднатися до існуючої команди»: ви отримуєте URL і повну копію історії, а не створюєте репозиторій з нуля.
}

::Clone — команда клонування::
Яка мінімальна команда клонує віддалений репозиторій у поточну директорію? {
=git clone <URL>
~git init --clone <URL>
~git remote clone origin
#### Після git clone Git сам створює папку проекту, .git і завантажує об'єкти. Окремий git init у цільовій папці не потрібен.
}

::Clone — куди потрапляють файли::
Чому після clone команда git branch у «батьківській» папці clone не показує гілок? {
=Потрібно перейти (cd) у вкладену папку репозиторію, де є .git
~Клонування не копіює гілки, лише файли
~Гілки з'являються лише після git push
#### У демо помилка очікувана: корінь clone/ — просто контейнер. Справжній репозиторій — у sequel-practice (або git-курс), куди й треба зайти.
}

::Clone — tracking branches::
Що додатково показує git branch -a після клонування? {
=Локальні гілки та віддалені tracking-гілки (наприклад remotes/origin/main)
~Лише приховані паролі SSH
~Список усіх форків на GitHub
#### Клон створює зв'язок з origin і remote-tracking гілки. Вони допомагають бачити, що є на GitHub, навіть якщо локально checkout лише main.
}

::Clone — git branch -vv::
Навіщо використовують git branch -vv у відео? {
=Щоб побачити, яка локальна гілка прив'язана до якої віддаленої (upstream)
~Щоб видалити всі віддалені гілки
~Щоб перейменувати origin на github
#### Після налаштування upstream для feature поруч із локальною гілкою з'являється [origin/feature] — видно пару local↔remote.
}

::Clone — історія комітів::
Що дає git log --oneline одразу після clone? {
=Компактний перегляд усієї історії комітів, яку завантажили з віддаленого репозиторію
~Лише список файлів без комітів
~Прогноз майбутніх конфліктів
#### Клон тягне не лише файли, а й .git з повною історією — log одразу показує попередні коміти проекту.
}

::Clone — push нової гілки::
Після створення feature і commit чому знову потрібен git push --set-upstream origin feature? {
=Локальна feature ще не пов'язана з жодною гілкою на origin
~HTTPS-клонування забороняє push у main
~git clone видаляє origin після першого коміту
#### Нова локальна гілка не існує на сервері. --set-upstream публікує її і задає tracking — так само, як у уроці про конфлікти.
}

::Clone — HTTPS і авторизація::
Чому при push після HTTPS-клону з'являється вікно входу через браузер? {
=HTTPS не використовує налаштовану SSH-пару і потребує окремої авторизації GitHub
~Браузер потрібен лише для git log
~SSH-ключі автоматично блокуються після clone
#### У цьому уроці свідомо обрали HTTPS-URL. Тому push запитує логін через браузер, на відміну від SSH-сценарію з попередніх відео.
}

::Clone — fork проти clone::
Чим fork на GitHub відрізняється від git clone за змістом уроку? {
=Fork створює вашу незалежну копію репозиторію в профілі GitHub; clone копіює репозиторій локально на диск
~Fork і clone — це одна й та сама команда в терміналі
~Fork завантажує лише README без .git
#### Наприкінці узагальнюють опції: fork — копія на сервісі у вашому акаунті, clone — робоча копія на машині для комітів і push.
}

::Clone — замикання циклу::
Як після merge PR локально отримати файл з main, який з'явився на GitHub? {
=Перейти на main (git checkout main) і виконати git pull
~Видалити .git і знову клонувати весь репозиторій
~Зробити лише git fetch без checkout
#### У демо після merge файл є на origin/main, але не в локальній main. Checkout main + pull підтягує злитий результат — це завершення циклу clone→feature→PR→sync.
}
