::Commit проти Saved::
У відео порівнюють звичайне збереження файлу (Ctrl+S / Saved) із git commit. Яка ключова перевага коміту згідно з лекцією? {
=Git фіксує не лише факт збереження, а й хто, коли і що саме змінив — до рівня рядків і слів
~Git автоматично створює резервну копію файлу на зовнішньому диску при кожному коміті
~Коміт замінює потребу в будь-яких повідомленнях і метаданих, на відміну від Saved
#### Коміт у Git — це детальний запис змін: автор, час і точний зміст правок. Saved у текстовому редакторі лише перезаписує поточну версію файлу без такої історії.
}

::Перша практична перевага Git::
Яку першу важливу перевагу Git як інструменту контролю версій наголошує лектор на початку теоретичного блоку? {
=Можливість швидко повернутися до будь-якої збереженої версії проекту
~Автоматичне злиття змін від усіх учасників без жодних конфліктів
~Повна заміна командного рядка графічним інтерфейсом Microsoft Word
#### Коли зміни щось ламають або виникають конфлікти, Git дозволяє відкотитися до попередньої робочої версії — це перша названа перевага.
}

::Гілки та merge у теорії::
Навіщо в лекції вводять поняття гілок і merge на прикладі одного текстового файлу? {
=Щоб два спеціалісти могли паралельно працювати в ізольованих середовищах і потім безпечно об'єднати зміни
~Щоб кожен коміт автоматично дублювався у хмарі GitHub без додаткових команд
~Щоб замінити staging area окремими копіями файлу на робочому столі
#### Гілки дають окремі «лінії розвитку» (V1→V2→V3), а merge зшиває результати — так команда не блокує одна одну під час роботи над тим самим проектом.
}

::Репозиторій як проект::
Як лектор уточнює об'єкт, яким керує Git, переходячи від одного файлу до реального проекту? {
=Репозиторій — це папка проекту з багатьма файлами та підпапками, а не один текстовий документ
~Репозиторій — це лише прихована папка .git без зв'язку з файлами користувача
~Репозиторій існує тільки після першого push на GitHub
#### Git відстежує зміни у всій структурі проекту (код, дані, матеріали), до якої ви надали йому доступ через ініціалізацію.
}

::Збереження незмінених файлів::
Як Git поводиться з файлом, який не змінювався між комітами? {
=Зберігає посилання на наявну версію, а не повну копію файлу щоразу
~Видаляє файл з історії, доки хтось знову його не відредагує
~Автоматично копіює файл у staging area при кожному коміті
#### Git ефективний завдяки дельта-подібному підходу: незмінені файли не дублюються, а посилаються на вже збережену версію.
}

::Стан modified::
Що означає статус modified у моделі Git, описаній у цьому відео? {
=Файл відредаговано у робочій зоні, але ще не пройшов повний цикл збереження через staging і commit
~Файл уже закомічений і є останньою версією в репозиторії
~Файл видалено з диска і більше не існує в проекті
#### Modified — це «чернетка в роботі»: зміни є локально, але Git ще не зафіксував їх як нову версію проекту.
}

::Роль staging area::
Яку функцію виконує staging area (проміжна ланка) у циклі роботи з Git? {
=Дозволяє обрати, які саме зміни потраплять у наступний коміт, перед остаточним збереженням
~Зберігає остаточну історію всіх комітів замість папки .git
~Синхронізує репозиторій із віддаленим сервером автоматично
#### Staging — це «чернетка коміту»: ви спочатку додаєте туди потрібні файли (git add), і лише потім фіксуєте версію (git commit).
}

::Дві команди замість Ctrl+S::
Чим цикл Git принципово відрізняється від одного натискання Save у редакторі? {
=Потрібні дві команди: спочатку додати зміни до staging (git add), потім зафіксувати коміт (git commit)
~Достатньо однієї команди git save, яка одразу публікує код у Main
~Зміни зберігаються лише після перезавантаження комп'ютера
#### Розділення add і commit дає контроль над тим, що саме входить у версію — це основа гнучкого менеджменту змін і вирішення конфліктів.
}

::Метадані коміту::
Яку інформацію Git гарантовано зберігає про кожен коміт згідно з лекцією? {
=Хто зробив зміну, коли, і що саме було додано або видалено
~Лише назву файлу без порівняння вмісту між версіями
~Пароль автора та IP-адресу комп'ютера
#### На відміну від простого Save, коміт — це структурований запис змін із контекстом, незамінний для командної роботи та аудиту.
}

::Охоплення всього проекту::
Який висновок лектор робить щодо «поля зору» Git у репозиторії? {
=Git працює одночасно з усіма відстежуваними файлами проекту — ніщо не лишається поза увагою, якщо ви це не виключили
~Git відстежує лише один активний файл, який відкритий у редакторі
~Git ігнорує вкладені папки й працює тільки з кореневим каталогом
#### Після ініціалізації репозиторію Git контролює зміни по всьому проекту, що критично для цілісності версій.
}
