::Push — сценарій першого зв'язку::
У відео розбирається ситуація: локальний проект уже є, а репозиторій на GitHub — порожній. Який наступний крок після git init і initial commit? {
=git remote add origin <ssh-url> і далі git push
~git clone origin у ту саму папку поверх існуючих файлів
~Видалити локальний .git і заново завантажити архів з GitHub
#### GitHub пропонує саме цей сценарій: ініціалізувати локально, прив'язати origin до віддаленого URL і штовхнути код. Це перший сценарій «локальний проект → порожній remote».
}

::Push — роль origin::
Навіщо в команді git remote add origin використовують слово origin? {
=Це псевдонім віддаленого репозиторію, на який далі посилаються git push і git pull
~Це назва гілки за замовчуванням замість main
~Це обов'язковий логін адміністратора GitHub
#### У демо origin — коротке ім'я для URL test.git. Потім достатньо git push origin main: Git знає, куди саме відправляти зміни.
}

::Push — SSH проти HTTPS::
Чому в курсі для push обрано SSH-URL замість HTTPS? {
=Після налаштування SSH-ключів не потрібно щоразу вводити пароль при відправці коду
~HTTPS не дозволяє push у публічні репозиторії
~SSH автоматично створює README і initial commit на GitHub
#### Інструктор прямо порівнює: з HTTPS довелось би вводити пароль при кожному push. SSH-пара вже налаштована — тому для демо це зручніший і безпечніший варіант.
}

::Push — перший push::
Що робить перший успішний git push у сценарії з порожнім віддаленим репозиторієм? {
=Передає локальні коміти та файли на GitHub, синхронізуючи історію
~Лише копіює назву гілки, але не передає файли
~Автоматично створює pull request і зливає гілки
#### Після remote add origin push «штовхає» локальний initial commit і вміст проекту на GitHub. Саме цього й чекали від порожнього test-репозиторію.
}

::Push — git pull після merge PR::
Після merge pull request на GitHub локальна main відстає. Яку команду інструктор використовує, щоб підтягнути оновлення? {
=git pull
~git fetch --delete origin
~git checkout -b main
#### У відео підкреслено: віддалений репозиторій стає джерелом актуальної версії. git pull затягує зміни з GitHub і поєднує їх із локальною main — зворотна операція до push.
}

::Push — віддалений репозиторій як джерело правди::
Чому після командної роботи важливо регулярно оновлювати локальне середовище? {
=Віддалений репозиторій може містити злиті зміни інших учасників і бути попереду локальної копії
~GitHub автоматично видаляє локальні гілки старші за 7 днів
~Локальний репозиторій завжди новіший за віддалений після git init
#### Інструктор показує: після PR на GitHub локальна feature/main не містить останніх змін. Без pull ви працюєте з застарілою версією коду.
}

::Push — tracking branches::
Що з'являється локально після підключення віддаленого репозиторію, хоча git branch їх не показує звичайним чином? {
=Віддалені tracking-гілки (наприклад origin/main), які відображають стан GitHub
~Дублікати всіх файлів у папці origin/
~Нові SSH-ключі для кожної гілки
#### У транскрипті згадано tracking branches: з git branch -a видно не лише локальні гілки, а й віддалені. Спочатку оновлюються вони, потім — ваша робоча гілка.
}

::Push — git pull як fetch + merge::
Як у відео описують суть git pull для гілки main? {
=Команда одночасно підтягує зміни з віддаленого репозиторію і зливає їх із локальною main
~Команда лише показує diff, але не змінює файли
~Команда видаляє локальні незакомічені зміни
#### pull — «зворотний push»: затягує актуальний код з GitHub і робить merge з вашою main, якщо конфліктів немає. Саме так локальне середовище наздоганяє remote.
}

::Push — перейменування master на main::
GitHub у інструкціях пропонує перейменувати гілку на main. Чому в демо цей крок пропускають? {
=Локальний репозиторій уже створено з гілкою main, тож додаткове перейменування не потрібне
~GitHub забороняє використовувати main у публічних репозиторіях
~Команда git branch -m main видаляє історію комітів
#### Інструктор зазначає: вони з початку працювали з main, тому крок із master→main з інструкцій GitHub для них зайвий.
}

::Push — практика циклу::
Яку пораду дає інструктор перед переходом до наступної теми? {
=Кілька разів пройти цикл: змінити локально → push → PR/merge → pull
~Використовувати лише GitHub Desktop і не повторювати команди
~Більше ніколи не переходити на main після feature-гілок
#### Наприкінці уроку пропонується свідомо відпрацювати повний цикл синхронізації — це закріплює push, pull і розуміння того, хто «попереду»: local чи origin.
}
