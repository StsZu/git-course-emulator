::Чисте дерево перед гілками::
Чому перед створенням гілок лектор перевіряє відсутність незакомічених змін? {
=Найкраща практика — мати чисте робоче дерево, щоб не перенести «зайве» в нову гілку
~Git не дозволяє створювати гілки, якщо є хоч один файл у проекті
~Без clean status git branch завжди видає помилку merge
#### Незакомічені зміни «подорожують» з вами між гілками — чистий старт зменшує сюрпризи.
}

::git branch без перемикання::
Що відбувається після git branch feature1 без checkout? {
=Гілка feature1 створюється, але ви залишаєтесь на поточній гілці (Main)
~Ви автоматично переходите на feature1 і HEAD зміщується
~Команда видаляє гілку Main як застарілу
#### branch лише додає ім'я нової лінії; перемикання — окремий крок (checkout або switch).
}

::git checkout feature1::
Яка мета git checkout feature1 у практичній вправі? {
=Перемкнути робочий контекст на гілку feature1, щоб коміти йшли вже в неї
~Скопіювати папку .git на робочий стіл
~Об'єднати feature1 у Main без перевірки змін
#### Checkout змінює активну гілку: історія та HEAD тепер належать feature1.
}

::git checkout -b::
Чим зручна команда git checkout -b feature2? {
=Одночасно створює гілку і перемикає на неї — два кроки в одному
~Видаляє попередню гілку після створення нової
~Публікує гілку на GitHub без налаштування remote
#### -b (branch) — улюблений скорочений workflow: «створив і одразу працюю».
}

::Історія нової гілки::
Що показує git log --oneline одразу після переходу на нову гілку? {
=Повну історію комітів до точки відгалуження — гілка успадковує минуле Main
~Порожній лог без жодного коміту
~Лише коміти, зроблені після merge
#### Нова гілка не «з нуля»: вона пам'ятає всі коміти до моменту створення.
}

::git merge з Main::
З якої гілки потрібно виконувати git merge feature1, щоб перенести зміни в основну? {
=Перебуваючи на Main (цільовій гілці), виконати git merge feature1
~Перебуваючи на feature1, виконати git merge Main
~З будь-якої папки поза репозиторієм
#### Merge завжди «вливає» джерело в ту гілку, на якій ви зараз стоїте (HEAD).
}

::Ізоляція змін між гілками::
Що демонструє приклад: зміни в feature1 (видалення тексту) vs повернення на Main? {
=Зміни в feature1 не видно на Main, доки не виконано merge — гілки ізольовані
~Зміни миттєво з'являються на всіх гілках автоматично
~Main завжди копіює файли з feature1 при кожному git status
#### Ізоляція — суть гілок: експеримент у feature не ламає стабільний Main.
}

::git branch -d::
Коли можна безпечно видалити гілку git branch -d feature1? {
=Після merge, коли всі зміни feature1 уже інтегровані в Main
~Одразу після git branch feature1, не роблячи жодного коміту
~Лише після видалення папки .git
#### -d (delete) перевіряє, що гілка злита; інакше Git попередить, що зміни буде втрачено.
}

::Конфлікт при merge::
Яка ситуація в практиці призводить до конфлікту merge (feature3)? {
=Різні зміни того самого файлу в Main і feature3 — Git не може вибрати варіант автоматично
~Дві гілки з різними іменами завжди конфліктують незалежно від файлів
~Конфлікт виникає лише при відсутності інтернету
#### Конфлікт — запрошення вирішити суперечність вручну перед завершенням merge.
}

::HEAD після merge::
Що відбувається з HEAD після успішного merge feature у Main? {
=HEAD вказує на новий merge-коміт (або fast-forward), що включає зміни з feature
~HEAD зникає з git log --oneline
~HEAD переміщується на віддалений сервер
#### Після злиття актуальна точка Main «підтягується» до об'єднаного стану проекту.
}
