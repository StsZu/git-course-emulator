::git status після git init::
Що показує git status одразу після ініціалізації репозиторію, коли комітів ще не було? {
=Список невідстежуваних (untracked) файлів і папок, які Git ще не контролює
~Повідомлення «working tree clean» і відсутність будь-яких файлів
~Автоматично створений initial commit із усіма файлами проекту
#### Після git init Git бачить файли, але не відстежує їх, доки ви явно не додасте їх через git add.
}

::Порожня папка Data sources::
Чому git status спочатку показує лише дві папки (code, materials), хоча в проекті є три? {
=Git не відстежує порожні каталоги — папка Data sources на той момент не містила файлів
~Git ігнорує папки з пробілами в назві за замовчуванням
~Папка Data sources автоматично потрапляє в .gitignore при ініціалізації
#### Git працює з файлами, а не з «пустими» директоріями: поки в каталозі немає вмісту, він не з'явиться в статусі.
}

::git add .::
Що робить команда git add . при підготовці першого коміту? {
=Додає всі невідстежувані та змінені файли в staging area
~Створює нову гілку з іменем поточної директорії
~Видаляє з репозиторію файли, які не вказані явно
#### Крапка означає «усі зміни в поточному каталозі та нижче» — зручно для initial commit, хоча на практиці частіше додають файли вибірково.
}

::Обов'язкове повідомлення коміту::
Як у відео виконується перший коміт із повідомленням? {
=git commit -m "initial commit" — повідомлення обов'язкове, без нього коміт не створюється
~git commit без параметрів автоматично генерує повідомлення «initial commit»
~git push -m "initial commit" — коміт і відправка виконуються однією командою
#### Повідомлення коміту — обов'язкова частина фіксації версії; воно пояснює, що саме змінилося.
}

::Working tree clean::
Що означає рядок «working tree clean» після успішного коміту? {
=У робочій директорії немає незакомічених змін порівняно з останнім комітом
~Репозиторій видалено з диска і потрібна повторна ініціалізація
~Усі файли проекту автоматично відправлені на GitHub
#### Clean working tree — «чистий стіл»: усе, що ви змінили, уже збережено в останньому коміті.
}

::Дві зони в git status::
Які дві зони одночасно відображає git status згідно з практикою в цьому відео? {
=Робочу зону (working area / working tree) і staging area
~Лише віддалений репозиторій на GitHub і локальну копію .git
~Тільки гілки Main і Master без файлів
#### git status — головний «дашборд»: він показує, що змінено локально і що вже підготовлено до коміту.
}

::Git з будь-якої папки репозиторію::
Чому в лекції можна виконувати git-команди, перебуваючи в підпапці materials, а не лише в корені my_project? {
=Git відстежує файли в усьому репозиторії, незалежно від поточної робочої директорії в терміналі
~Git Bash автоматично переміщує вас у корінь при кожній команді
~Команди працюють лише в підпапці, де був виконаний git init
#### Репозиторій — єдине ціле: git status, add і commit коректно бачать зміни з будь-якої вкладеної папки.
}

::git log::
Навіщо в практичній частині використовують git log після серії комітів? {
=Щоб переглянути історію комітів із унікальними ідентифікаторами (хешами) і повідомленнями
~Щоб видалити всі попередні коміти й почати історію заново
~Щоб автоматично створити нову гілку для кожного запису в логу
#### git log — хронологія версій проекту; кожен запис має ID, за яким можна звернутися до конкретного стану.
}

::git diff::
Що показує звичайна команда git diff у прикладі з відео? {
=Різницю між робочою директорією та останнім комітом (незбережені зміни)
~Різницю між двома віддаленими гілками на GitHub
~Список усіх файлів у staging area без порівняння вмісту
#### git diff без опцій відповідає на питання: «Що я змінив локально, але ще не закомітив?»
}

::Колір статусу після git add::
Як змінюється вивід git status після git add для файлів першого коміту? {
=Файли, готові до коміту, відображаються в staging area (у відео — зелені посилання замість червоних untracked)
~Файли зникають з виводу, бо Git вважає їх видаленими
~Статус залишається untracked, доки не виконано git push
#### Після add файли переходять у staged — вони «в черзі» на наступний коміт, і status це чітко показує.
}
