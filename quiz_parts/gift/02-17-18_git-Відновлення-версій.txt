::git restore за замовчуванням::
Що відбувається при git restore test.txt без додаткових опцій після випадкового видалення файлу? {
=Файл відновлюється з останнього коміту в робочу директорію
~Створюється нова гілка з іменем test.txt
~Видаляється останній коміт разом із усім проектом
#### За замовчуванням restore бере зміст із HEAD (останнього збереженого стану гілки).
}

::Видалення і git status::
Як Git реагує на видалення файлу rm test.txt, який був у останньому коміті? {
=Фіксує видалення як зміну; git status показує, що файл deleted
~Ігнорує видалення, бо файл існує в історії назавжди на диску
~Автоматично створює backup на Google Drive
#### Видалення у робочій зоні — теж зміна; її можна закомітити або скасувати через restore.
}

::git restore --source=::
Навіщо в прикладі використовують git restore --source=<commit-id> test.txt? {
=Щоб відновити версію файлу з конкретного коміту, а не лише з останнього
~Щоб завантажити файл із інтернету за посиланням commit-id
~Щоб перейменувати гілку Main у test.txt
#### Коли потрібен «старий» вміст, source вказує точку в історії, звідки брати файл.
}

::git show::
Яку роль виконує git show <commit-id> у сценарії відновлення? {
=Показує, що саме змінилося в тому коміті, щоб обрати потрібну версію файлу
~Видаляє коміт із історії без можливості відкату
~Встановлює Git на новий комп'ютер
#### show — «рентген» коміту: бачите зміни в test.txt і вирішуєте, до якого стану повернутися.
}

::git log --oneline::
Чому в практиці зручно використовувати git log --oneline? {
=Дає компактний перелік комітів із короткими ID та повідомленнями для швидкого пошуку
~Повністю замінює git status і більше не потрібен
~Показує лише коміти поточного тижня, старіші приховує
#### --oneline стискає історію до «шпаргалки» — зручно копіювати ID для restore --source.
}

::HEAD у git log::
Що означає позначення HEAD -> main біля верхнього коміту в git log? {
=HEAD вказує на найсвіжіший коміт поточної гілки main
~HEAD означає, що репозиторій пошкоджено і потрібен git init
~HEAD — це ім'я файлу, який зараз відкритий у Notepad
#### HEAD — «ви тут» на карті історії: актуальний стан гілки, з яким порівнюється working tree.
}

::git reset::
Для чого в кінці відео вводять git reset (з опціями soft/hard)? {
=Щоб відкотити весь репозиторій до попереднього коміту, а не лише один файл
~Щоб збільшити шрифт у терміналі Git Bash
~Щоб створити pull request на GitHub
#### reset зміщує HEAD (і за потреби working tree/index) — це «машина часу» для всього проекту; з ним треба обережно.
}

::git reset --hard::
Чим небезпечний git reset --hard згідно з попередженням лектора? {
=Може безповоротно скинути незакомічені зміни в робочій директорії
~Автоматично видаляє віддалений репозиторій на GitHub
~Блокує обліковий запис GitHub на 30 днів
#### --hard підганяє і коміт, і файли на диску — незбережена робота може зникнути.
}

::restore vs checkout::
Чому для відновлення файлів у курсі акцентують git restore, а не checkout? {
=restore призначений для відновлення файлів; checkout у подальшому використовують переважно для гілок
~restore працює лише на Windows, checkout — лише на Linux
~Це дві назви однієї команди без жодної різниці в поведінці
#### Розділення ролей зменшує плутанину: restore — файли, checkout/switch — контекст гілки.
}

::git rm::
Чим git rm test.txt відрізняється від звичайного rm test.txt у відео? {
=git rm одразу реєструє видалення для Git і готує зміну до staging area для коміту
~git rm видаляє файл лише з історії, залишаючи копію на диску назавжди
~git rm працює тільки в VS Code, а не в Git Bash
#### git rm інтегрує видалення в робочий цикл Git: зміну можна закомітити або скасувати через restore.
}
