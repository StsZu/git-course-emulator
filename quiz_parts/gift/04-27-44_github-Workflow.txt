::Локальні зони Git::
Які три ключові зони локального Git нагадує автор на схемі workflow? {
=Working Area (Working Tree), Staging Area та локальний репозиторій після commit
~Cloud Area, Email Area та Fork Area
~Terminal, Browser і Desktop без commit
#### Класичний локальний цикл: редагуєш → staging → commit у локальну базу.
}

::Команда для staging::
Яку команду використовують, щоб підготувати зміни до коміту? {
=git add
~git push
~git clone
#### git add переносить зміни з робочої зони в staging area перед фіксацією комітом.
}

::Команда для збереження версії::
Як зміни потрапляють у локальний репозиторій після staging? {
=Через git commit
~Через git pull
~Через збереження файлу в VS Code без Git
#### Commit записує підготовлені зміни в історію локального репозиторію.
}

::Віддалений репозиторій у workflow::
Яку роль відіграє віддалений репозиторій GitHub у узагальненій схемі? {
=Сховище коду й комітів у хмарі, куди push і звідки pull синхронізують команду
~Тимчасовий кеш лише для README
~Заміну локальної папки .git
#### Remote — спільна точка синхронізації; локальна робота лишається на машині розробника.
}

::git push::
Коли в workflow застосовують git push? {
=Після локальних комітів, коли потрібно відправити гілку або зміни на віддалений репозиторій
~Перед git add, щоб завантажити файли з інтернету
~Лише після видалення всіх локальних гілок
#### Push публікує ваші напрацювання на GitHub, роблячи їх доступними команді.
}

::git pull::
Навіщо потрібен git pull у командному workflow? {
=Щоб підтягнути оновлення з віддаленого репозиторію, коли локальна копія відстала
~Щоб створити новий GitHub-акаунт
~Щоб видалити останній коміт без сліду
#### Pull синхронізує вас з хмарою, де колеги вже опублікували нові зміни.
}

::Продакшн-версія::
Що стає «опорною» версією коду в командному процесі згідно з уроком? {
=Віддалений репозиторій, на який можна спиратися при подальшому розвитку інфраструктури
~Будь-яка несохранена копія на робочому столі
~Лише перший коміт у локальній історії
#### У хмарі збирається перевірений стан, з яким вирівнюються інші учасники.
}

::Гілка та PR у повному циклі::
Як виглядає типовий цикл після push гілки на віддалений сервер? {
=Відкрити Pull Request, отримати рев'ю, після merge інші підтягують main через git pull
~Автоматично видалити main і почати проєкт заново
~Заборонити pull усім учасникам команди
#### Повний remote-workflow: гілка → push → PR → merge → pull оновленої main.
}

::Малюнки схем::
Яку пораду дає автор щодо засвоєння нових Git-команд? {
=Малювати схеми взаємодії команд із зонами Git при вивченні нових задач
~Запам'ятовувати команди без розуміння контексту
~Використовувати лише веб-інтерфейс GitHub без терміналу
#### Візуалізація зон і потоків допомагає не плутати add, commit, push і pull.
}

::Помилки як навчання::
Як ставитися до помилок під час налаштування зв'язку Git і GitHub? {
=Це нормальна частина практики; варто аналізувати помилки й звертатися до документації чи інструментів
~Негайно видалити Git і відмовитися від GitHub
~Ігнорувати помилки, бо вони зникнуть після reboot
#### Автор прямо заохочує проходити через типові проблеми — це найміцніший спосіб засвоєння workflow.
}
