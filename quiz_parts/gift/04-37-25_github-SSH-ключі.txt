::SSH — навіщо потрібна пара ключів?::
Якщо ви вже налаштували SSH для GitHub, що саме забезпечує безпечний обмін між локальною машиною та віддаленим репозиторієм? {
=Пара SSH-ключів: приватний ключ на комп'ютері та відповідний публічний ключ у профілі GitHub
~Одноразовий пароль від GitHub, який зберігається в README проекту
~Логін і пароль HTTPS, які Git запам'ятовує після першого push
#### У відео підкреслено: доступ тримається на парі ключів — локально зберігається приватна частина, а на GitHub додається публічна. Саме збіг цієї пари дозволяє автентифікувати push/pull без повторного введення пароля.
}

::SSH — де зберігається публічний ключ?::
Після генерації SSH-ключа на локальній машині куди потрібно додати публічну частину, щоб GitHub «впізнавав» ваш комп'ютер? {
=У налаштуваннях акаунта GitHub, у секції SSH keys
~У файл .gitignore кожного репозиторію
~У локальний файл config усередині папки .git
#### Інструктор показує шлях Settings → SSH keys і вставляє туди вміст публічного файлу, скопійований через cat. GitHub не генерує ключ за вас — ви створюєте його локально і лише публічну частину реєструєте на сервісі.
}

::SSH — генерація ключа::
Яку команду використовують на початку налаштування SSH-зв'язку з GitHub? {
=ssh-keygen з email, прив'язаним до акаунта GitHub
~git remote add origin з HTTPS-посиланням
~ssh-add github.com для автоматичного створення ключа
#### У конспекті курсу перший крок — ssh-keygen; email слугує міткою ключа і має відповідати вашому GitHub-акаунту. Саме ця команда створює файли приватного та публічного ключів.
}

::SSH — стандартний шлях збереження::
Під час ssh-keygen інструктор натискає Enter на запит про файл для збереження ключа. Що це означає на практиці? {
=Ключ буде збережено у стандартному розташуванні, яке Git і ssh очікують за замовчуванням
~Ключ не буде збережено на диск і зникне після закриття терміналу
~Ключ автоматично завантажиться в GitHub без ручного копіювання
#### Enter приймає шлях у дужках — типову директорію ~/.ssh. Це зручно, бо наступні команди (ssh-add, cat публічного файлу) теж посилаються на стандартні імена файлів.
}

::SSH — passphrase::
Що відбувається, якщо двічі натиснути Enter на етапі passphrase під час ssh-keygen? {
=Приватний ключ створюється без додаткового пароля на ключ
~GitHub автоматично відхилить такий ключ як небезпечний
~Потрібно буде вводити пароль GitHub при кожному git push
#### У демо інструктор свідомо лишає passphrase порожнім: для навчання цього достатньо. Пара ключів уже забезпечує автентифікацію; passphrase — додатковий рівень захисту самого приватного файлу.
}

::SSH — роль ssh-agent::
Навіщо після генерації ключа перевіряють, що ssh-agent запущений, і додають ключ через ssh-add? {
=Щоб агент тримав приватний ключ у пам'яті сесії і Git міг ним користуватися без повторного введення
~Щоб публічний ключ автоматично з'явився в профілі GitHub
~Щоб замінити необхідність git init у локальному проекті
#### Після ssh-keygen ключ існує як файл; ssh-add передає його агенту. Це типовий ланцюжок: агент працює — ключ додано — SSH-з'єднання з GitHub проходить без зайвих запитань.
}

::SSH — як отримати публічний ключ для GitHub::
Як інструктор переглядає вміст публічного ключа перед копіюванням у GitHub? {
=Командою cat для файлу публічного ключа
~Командою git status у корені репозиторію
~Через вкладку Branches у вебінтерфейсі GitHub
#### У відео показано cat (у транскрипті — cet) для читання згенерованого файлу. Саме цей текст і потрібно вставити в поле нового SSH key на GitHub.
}

::SSH — тест з'єднання::
Якою командою перевіряють, що локальна машина успішно автентифікується на GitHub через SSH? {
=ssh -T git@github.com
~git push origin main --dry-run
~ssh-keygen --test github.com
#### Після додавання ключа інструктор тестує зв'язок саме цією командою. Успішна відповідь означає, що GitHub впізнав ваш публічний ключ — далі можна push/pull по SSH без паролів.
}

::SSH — перше підключення до хоста::
Що потрібно зробити, якщо при першому ssh -T git@github.com система питає про додавання github.com до known hosts? {
=Підтвердити з'єднання (yes), бо це разове додавання хоста до списку довірених на вашому комп'ютері
~Скасувати і одразу видалити згенерований ключ
~Перейти на HTTPS, бо SSH «не підтримується» GitHub
#### У транскрипті пояснюється: Git/SSH перевіряє, чи ви вже довіряєте цьому хосту. Підтвердження yes зберігає відбиток сервера локально — це нормальна поведінка при першому контакті.
}

::SSH — безпека ключа::
Чому інструктор попереджає не поширювати приватний ключ і планує видалити демо-ключ після уроку? {
=Будь-хто з доступом до вашої пари ключів може працювати з вашими віддаленими репозиторіями від вашого імені
~GitHub блокує акаунт, якщо ключ старіший за 24 години
~Приватний ключ потрібен лише для git clone, а не для push
#### У відео прямо сказано: хто має цей ключ — той отримує доступ до віддаленого середовища. Тому приватну частину тримають лише на своїй машині, а публічну — обережно додають лише у свій профіль.
}
