::Конфлікти — push нової гілки::
При першому push гілки feature виникає помилка про відсутність upstream. Яке рішення показано у відео? {
=git push --set-upstream origin feature
~git pull origin main --force
~git branch -D feature і повторний commit у main
#### Git підказує встановити upstream: локальна feature ще не пов'язана з віддаленою. Команда з --set-upstream створює гілку на GitHub і прив'язує її до локальної.
}

::Конфлікти — звичайний workflow::
Який порядок дій у «здоровому» сценарії після коміту в feature? {
=Push гілки → pull request → merge → за потреби видалити feature
~Одразу merge feature в main локально без push
~Видалити main і перейменувати feature на main
#### У демо це стандартний командний workflow: feature публікується, PR об'єднує зміни в main на GitHub, після чого feature можна прибрати.
}

::Конфлікти — розходження історій::
Як у відео штучно створюють ситуацію конфлікту в main? {
=Локально змінюють index.html у main, а паралельно редагують той самий файл прямо на GitHub
~Видаляють .git і клонують репозиторій двічі
~Вимикають SSH і переходять на FTP
#### Конфлікт виникає, коли і локальна main, і віддалена main змінили один рядок по-різному (Local Main vs Remote Main). Це реалістична ситуація розходження історій.
}

::Конфлікти — відхилений push::
Чому git push відхиляється, коли віддалений репозиторій містить коміти, яких немає локально? {
=Push перезаписав би чужу роботу на GitHub, тому Git вимагає спочатку інтегрувати віддалені зміни
~SSH-ключі автоматично блокуються після merge PR
~Гілка main може існувати лише в одній копії — локально або на GitHub
#### Помилка push логічна: ви відстали від origin. Git захищає віддалену історію і пропонує спочатку підтягнути зміни, а не «штовхати» поверх них.
}

::Конфлікти — чому pull теж складний::
Що відбувається, коли після відхиленого push виконують git pull у конфліктній ситуації? {
=Git намагається злити зміни, але зупиняється на конфлікті в тому самому файлі
~Pull автоматично видаляє всі локальні коміти
~Pull створює нову гілку conflict без можливості merge
#### У демо pull не «магічно вирішує» проблему: він приносить Remote Main і стикається з Local Main у index.html — починається merge conflict.
}

::Конфлікти — вирішення у VS Code::
Як у відео завершують розв'язання конфлікту в редакторі? {
=Обирають, які зміни залишити (наприклад, приймають віддалену версію), і роблять commit злиття
~Закривають VS Code — Git сам випадково обирає версію
~Видаляють репозиторій і клонують заново без історії
#### VS Code показує локальну й віддалену версії; після вибору потрібен commit, що фіксує вирішення. Без цього репозиторій лишається в стані незавершеного merge.
}

::Конфлікти — після вирішення::
Що означає успішна синхронізація після розв'язання конфлікту? {
=Локальна main і origin/main знову вказують на узгоджену історію, і push проходить
~Всі feature-гілки автоматично видаляються з GitHub
~Конфліктні файли більше ніколи не можна редагувати
#### Після merge-коміту HEAD локальної main збігається з origin/main — саме цю синхронізацію показує кнопка Sync (pull, потім push) у VS Code.
}

::Конфлікти — ставлення до конфліктів::
Який головний меседж інструктора про merge conflicts? {
=Конфлікти — нормальна частина командної роботи; важливо розуміти, які зміни ви готові залишити
~Конфлікти означають, що репозиторій пошкоджено і його треба пересоздати
~Конфлікти виникають лише при використанні HTTPS
#### У підсумку сказано: боятися конфліктів не треба — треба аналізувати зміни, гілки й домовлятися в команді (або свідомо обирати версію самостійно).
}

::Конфлікти — комунікація в команді::
Чим практика з PR у відео відрізняється від реальної командної роботи? {
=У демо PR погоджують самостійно, а в команді це спільне рев'ю й обговорення
~У демо PR взагалі не потрібен
~У команді конфлікти неможливі через GitHub Desktop
#### Інструктор підкреслює: у навчанні все роблять solo, але в житті PR — це комунікація, перевірка коду й колективне рішення перед merge.
}

::Конфлікти — коли виникає push без upstream::
Ситуація з git push --set-upstream у цьому уроці пов'язана з чим? {
=Перша публікація нової локальної гілки, якої ще немає на віддаленому репозиторії
~Спробою push після вирішення конфлікту в main
~Переходом з SSH на HTTPS
#### Для нової feature GitHub ще не знає «пару» гілки. --set-upstream якраз створює віддалену feature і задає зв'язок для наступних push/pull.
}
