Якщо ви вже налаштували SSH для GitHub, що саме забезпечує безпечний обмін між локальною машиною та віддаленим репозиторієм? {
  =Пара SSH-ключів: приватний ключ на комп'ютері та відповідний публічний ключ у профілі GitHub
  ~Одноразовий пароль від GitHub, який зберігається в README проекту
  ~Логін і пароль HTTPS, які Git запам'ятовує після першого push
  #### У відео підкреслено: доступ тримається на парі ключів — локально зберігається приватна частина, а на GitHub додається публічна. Саме збіг цієї пари дозволяє автентифікувати push/pull без повторного введення пароля.
}

::SSH — де зберігається публічний ключ?::
Після генерації SSH-ключа на локальній машині куди потрібно додати публічну частину, щоб GitHub «впізнавав» ваш комп'ютер? {
  =У налаштуваннях акаунта GitHub, у секції SSH keys
  ~У файл .gitignore кожного репозиторію
  ~У локальний файл config усередині папки .git
  #### Інструктор показує шлях Settings → SSH keys і вставляє туди вміст публічного файлу, скопійований через cat. GitHub не генерує ключ за вас — ви створюєте його локально і лише публічну частину реєструєте на сервісі.
}

::SSH — генерація ключа::
Яку команду використовують на початку налаштування SSH-зв'язку з GitHub? {
  =ssh-keygen з email, прив'язаним до акаунта GitHub
  ~git remote add origin з HTTPS-посиланням
  ~ssh-add github.com для автоматичного створення ключа
  #### У конспекті курсу перший крок — ssh-keygen; email слугує міткою ключа і має відповідати вашому GitHub-акаунту. Саме ця команда створює файли приватного та публічного ключів.
}

::SSH — стандартний шлях збереження::
Під час ssh-keygen інструктор натискає Enter на запит про файл для збереження ключа. Що це означає на практиці? {
  =Ключ буде збережено у стандартному розташуванні, яке Git і ssh очікують за замовчуванням
  ~Ключ не буде збережено на диск і зникне після закриття терміналу
  ~Ключ автоматично завантажиться в GitHub без ручного копіювання
  #### Enter приймає шлях у дужках — типову директорію ~/.ssh. Це зручно, бо наступні команди (ssh-add, cat публічного файлу) теж посилаються на стандартні імена файлів.

  =Приватний ключ створюється без додаткового пароля на ключ

::SSH — роль ssh-agent::

  =Щоб агент тримав приватний ключ у пам'яті сесії і Git міг ним користуватися без повторного введення
