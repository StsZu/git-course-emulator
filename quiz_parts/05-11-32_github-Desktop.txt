  ~Назву гілки main на master
  ~Протокол з HTTPS на FTP
  #### У демо за замовчуванням — Documents, але інструктор свідомо кладе guitкурс на Desktop. Шлях визначає, де ви далі відкриватимете проект у VS Code.
}

::Desktop — URL при клонуванні::
Який тип посилання використовує GitHub Desktop у показаному клонуванні? {
  =HTTPS-URL репозиторію
  ~Лише SSH-URL незалежно від налаштувань
  ~Локальний шлях file:// без мережі
  #### У відео прямо сказано: Desktop пропонує HTTPS, не SSH. Це важливо порівняти з попереднім уроком, де push налаштовували через SSH-ключі.
}

::Desktop — fetch і pull::
Чим git fetch відрізняється від git pull у поясненні інструктора? {
  =fetch завантажує зміни з origin, але не зливає їх у поточну гілку; pull = fetch + merge
  ~fetch видаляє локальні коміти, pull їх відновлює
  ~fetch працює лише в Desktop, pull — лише в терміналі
  #### Це ключове розуміння «під капотом»: fetch оновлює віддалені відомості, а pull ще й інтегрує їх у вашу main.
}

::Desktop — відстеження змін::
Що відбувається в Desktop після редагування README у зовнішньому редакторі? {
  =Клієнт автоматично показує змінений файл і diff між версіями
  ~Потрібно вручну запустити git init
  ~Зміни ігноруються, доки не виконати git fetch
  #### Після збереження файлу Desktop одразу реєструє модифікацію й показує, що саме видалено/додано — без окремого git add у UI.
}

::Desktop — спрощений staging::
Як Desktop спрощує крок staging порівняно з CLI? {
  =Дозволяє одразу створити commit із змін у working area без окремого акценту на git add
  ~Повністю забороняє комітити окремі файли
  ~Автоматично робить push після кожного збереження файлу
  #### Інструктор звертає увагу: окремого кроку «передати в staging area» у UI немає — для простих сценаріїв commit створюється прямо зі списку змін.
}

::Desktop — неопублікований commit::
Як Desktop показує, що коміт є локальним, але ще не на GitHub? {
  =Є підказка опублікувати / Push origin, коміт позначений як ще не відправлений
  ~Зникає вкладка History
  ~З'являється попередження про конфлікт SSH
  #### Після commit інструктор бачить стрілку вгору й кнопку Push origin — тобто історія локально вже є, але remote ще не оновлено.
}

::Desktop — push з Desktop::
Що робить кнопка Push origin у продемонстрованому прикладі? {
  =Відправляє локальний commit у віддалену main на GitHub
  ~Створює fork репозиторію в іншому акаунті
  ~Завантажує архів проекту на email
  #### Push origin — той самий змістовний push, що й у CLI, але в один клік: локальна main → origin/main.
}

::Desktop — чому CLI все одно потрібен::
Чому інструктор показує Desktop лише після блоку про команди? {
  =Щоб розуміти, що відбувається «під капотом», і свідомо обирати інструмент
  ~Бо Desktop не вміє клонувати репозиторії
  ~Бо GitHub офіційно заборонив термінал
  #### Desktop — зручна надбудова, але без базових знань git add/commit/push/fetch ви не контролюєте процес і не розберетеся в помилках.
}
