::Роль GitHub у робочому процесі::
Якщо команда вже веде локальний Git-репозиторій, але потрібне єдине «джерело правди» для всіх розробників, яка роль найточніше описує віддалений репозиторій на GitHub? {
  =Він виступає хабом коду, де опублікована в хмарі версія вважається продакшн-версією, з якою синхронізуються локальні копії
  ~Він повністю замінює локальний Git, тому коміти роблять лише через веб-інтерфейс
  ~Він зберігає тільки README та issues, а сам код залишається лише на машинах розробників
  #### Віддалений репозиторій не скасовує локальний Git, а доповнює його: push відправляє перевірені зміни в хмару, pull підтягує актуальний стан. Саме хмарна версія стає надійною опорою для команди.
}

::Команди синхронізації::
Які дві Git-команди, згідно з уроком, відкривають взаємодію між локальною машиною та віддаленим репозиторієм? {
  =git push і git pull
  ~git init і git status
  ~git branch і git merge
  #### Push «штовхає» локальні коміти на віддалений сервер, pull «тягне» оновлення до себе. Без цієї пари локальна та хмарна історії жили б окремо.
}

::Чому обрали GitHub::
Чому для курсу зупинилися саме на GitHub серед GitLab, Bitbucket та інших платформ? {
  =Він лишається одним із найпопулярніших рішень і особливо важливий для персонального портфоліо та поширення коду
  ~Бо лише GitHub підтримує приватні репозиторії безкоштовно
  ~Бо GitHub — єдиний сервіс, який взагалі працює з Git
  #### Git — локальний інструмент; GitHub, GitLab і Bitbucket — різні хостинги віддалених репозиторіїв. GitHub обрано через популярність і силу в публічній презентації проєктів.
}

::Локальний vs віддалений::
Що залишається правдою про Git після переходу до GitHub, згідно з вступним відео? {
  =Git як програмне забезпечення працює локально, а GitHub надає онлайн-сервіс віддаленого сховища
  ~Після підключення GitHub усі зміни автоматично зберігаються в хмарі без git add і git commit
  ~Локальний репозиторій більше не потрібен, бо вся історія живе лише на GitHub
  #### Базовий цикл add → commit лишається локальним. GitHub додає шар спільного доступу, а не замінює сам Git.
}

::Сценарій одного користувача::
Який мотив використання GitHub для одного розробника без команди найкраще відповідає логіці уроку? {
  =Резервне копіювання коду, демонстрація напрацювань і портфоліо для потенційних роботодавців
  ~Обов'язкова вимога GitHub перед будь-яким git init
  ~Заміна IDE під час написання коду
  #### Навіть соло-розробник отримує хмарну копію, публічну вітрину проєктів і зручний спосіб показати свій рівень.
}

::Сценарій команди::
Що відбувається на другій машині після того, як колега виконав git push у спільний репозиторій? {
  =Перший розробник може git pull, щоб підтягнути останні зміни й працювати з актуальною версією
  ~Git автоматично перезаписує всі локальні файли без жодної команди
  ~Друга машина блокується, доки автор не надішле архів поштою
  #### Віддалений репозиторій синхронізує команду: push публікує, pull оновлює локальну копію.
}

::Коли пушити в хмару::
Коли, згідно з уроком, доцільно відправляти код на віддалений репозиторій? {
  =Після того як зміни перевірені й ви впевнені в їхній якості
  ~Одразу після кожного збереження файлу в редакторі, без коміту
  ~Лише після видалення всіх локальних гілок
  #### На машинах можуть бути експериментальні гілки; у хмару йде свідомо перевірена версія.
}

::Fork і Pull Request на рівні ідей::
Як у вступі описано співвідношення Fork і Pull Request? {
  =Fork створює власну копію чужого репозиторію, а Pull Request пропонує автору об'єднати ваші зміни після перегляду
  ~Fork одразу змінює оригінальний репозиторій без дозволу автора
  ~Pull Request потрібен лише для видалення репозиторію
  #### Fork дає незалежний простір для експериментів; Pull Request — культурний і технічний міст назад до оригіналу.
}
