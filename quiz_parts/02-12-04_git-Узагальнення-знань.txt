::Working Area::
Що таке working area (working directory) у узагальненні з відео? {
=Папка проекту з усіма файлами та підпапками, де ви безпосередньо працюєте й редагуєте
~Прихована службова директорія .git, де зберігаються лише хеші
~Віддалений репозиторій на сервері GitHub
#### Working area — ваш «робочий стіл»: тут живуть реальні файли, які ви відкриваєте в редакторі.
}

::Папка .git::
Яку роль відіграє прихована папка .git після ініціалізації? {
=Перетворює звичайну папку проекту на Git-репозиторій і зберігає історію версій
~Містить лише кеш браузера для швидшого відкриття файлів
~Автоматично публікує проект у соціальних мережах
#### .git — «мозок» репозиторію: без неї папка — просто набір файлів, з нею — керований проект із історією.
}

::Staging Area як проміжна ланка::
Навіщо Git вводить staging area (index) у трикроковій моделі? {
=Це проміжна ланка для підготовки обраних змін перед фіксацією нової версії в коміті
~Це місце постійного архівного зберігання всіх видалених файлів
~Це папка для тимчасових файлів операційної системи Windows
#### Staging дає контроль: ви вирішуєте, які саме зміни увійдуть у наступний знімок версії.
}

::Зміни замість повних копій::
Що Git зберігає після initial commit згідно з узагальненням у відео? {
=Не повні копії всіх файлів щоразу, а насамперед зміни, які ви вносите
~Лише назви файлів без їхнього вмісту
~Скріншоти екрана після кожного збереження
#### Після першого знімка Git ефективно працює з дельтами — це основа швидкості й економії місця.
}

::Три зони Git::
Які три ключові зони називає лектор у підсумковій схемі роботи з Git? {
=Working directory, staging area і commits (коміти)
~Command line, PowerShell і Git Bash
~Main, master і production
#### Модель «робоча зона → індекс → репозиторій» — карта, за якою ви читаєте git status і плануєте дії.
}

::Хеш коміту::
Чим є унікальний ID (хеш) кожного коміту? {
=Ідентифікатор, розрахований на основі змісту змін; ним Git посилається на об'єкти, а не на імена файлів
~Випадкове ім'я користувача, яке Git присвоює при git init
~Номер версії Python, встановлений на комп'ютері
#### Хеш — «відбиток» коміту: за ним можна точно отримати стан проекту в той момент.
}

::git log як історія::
Як у відео описують зв'язок між git log і комітами? {
=Кожен запис у логу — це коміт; за ID можна дістати потрібну версію файлу чи проекту
~Лог містить лише помилки терміналу, а не версії файлів
~Лог автоматично видаляється після кожного merge
#### Історія комітів — часова вісь проекту; log дає навігацію по цій вісі.
}

::Коміт як набір змінених файлів::
Що представляє собою один коміт у узагальненій моделі? {
=Набір файлів, змінених у межах цієї фіксації версії
~Повна копія всіх папок комп'ютера користувача
~Лише метадані автора без прив'язки до файлів
#### Коміт — знімок «що змінилося в цьому кроці», а не обов'язково всього диска.
}

::Синоніми staging area::
Які альтернативні назви staging area згадуються в узагальненні? {
=Index area та індекс — індексовані файли передаються в індекс перед комітом
~Remote area та cloud buffer для синхронізації з GitHub
~Master branch і HEAD як єдині синоніми staging
#### Staging area, index і cache — різні імена однієї проміжної зони підготовки змін до коміту.
}

~Налаштування роутера MikroTik для домашньої мережі
#### Узагальнення замикає базовий цикл add/commit і логічно веде до «машини часу» — відкату та відновлення.
