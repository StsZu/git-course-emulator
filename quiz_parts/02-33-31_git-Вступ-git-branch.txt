QUIZ (GIFT):
::Призначення гілки Main::
Яку роль виконує основна гілка Main (master) у теоретичному вступі? {
=Це production-гілка: код уже використовують інші системи, тому експерименти тут ризиковані
~Це тимчасовий кошик для видалених файлів перед git restore
~Це гілка лише для зберігання .gitignore без коду
#### Main — «живий» продукт; зміни тут впливають на те, що вже працює у реальному середовищі.
}

::main vs master::
Як лектор пояснює різницю між назвами main і master? {
=Функціональної різниці немає — це різні традиції іменування основної гілки
~master зберігає лише текстові файли, main — лише код Python
~Перейти з master на main можна лише через переустановку Git
#### Main і master — синоніми за змістом; важлива роль гілки, а не напис на ярлику.
}

::Створення гілки feature::
Що відбувається при створенні гілки feature від третього коміту на Main? {
=Утворюється повна копія стану проекту з усією попередньою історією комітів, але з іншою назвою гілки
~Створюється порожня папка без жодного коміту в історії
~Основна гілка Main автоматично видаляється
#### Нова гілка стартує з того ж знімка, що й точка відгалуження, і накопичує власні коміти ізольовано.
}

::merge у теорії::
Що таке merge у контексті роботи з гілками? {
=Об'єднання змін із feature-гілки в основну після успішних тестів
~Видалення всіх комітів feature-гілки без переносу змін
~Автоматичне перейменування всіх файлів проекту
#### Merge інтегрує напрацювання з ізольованого середовища назад у Main, зберігаючи історію обох ліній.
}

::HEAD як вказівник::
Як у відео визначають HEAD? {
=Вказівник на поточний коміт і гілку — «де ми зараз» в історії репозиторію
~Секретний пароль для push на GitHub
~Ім'я останнього файлу, відкритого в VS Code
#### HEAD рухається з вами: при нових комітах, перемиканні гілок або checkout на старий коміт.
}

::HEAD на різних гілках::
Що ілюструє порівняння HEAD на Main і HEAD на feature після кількох комітів у feature? {
=HEAD feature «попереду», бо містить новіші коміти; HEAD main лишається на старішому стані, доки не буде merge
~Обидва HEAD завжди вказують на один і той самий коміт незалежно від гілки
~HEAD на main автоматично видаляє коміти feature
#### Гілки розходяться: кожна має власний «кінчик» історії, поки ви їх не з'єднаєте.
}

::detached HEAD::
Коли виникає стан detached HEAD? {
=Коли checkout переводить на конкретний коміт без активної гілки — ви «відчеплені» від гілки
~Коли інтернет-з'єднання втрачено під час git push
~Коли папка .git перевищила 1 ГБ
#### У detached HEAD зміни не належать гілці; їх легко «втратити», якщо не знати, як повернутися.
}

::rebase vs merge::
Чим rebase відрізняється від merge у схемі з відео? {
=Rebase переносить коміти feature «після» HEAD main, вирівнюючи історію лінійно; merge зберігає розгалуження
~Rebase видаляє основну гілку, merge створює дві копії .git
~Rebase працює лише в VS Code, merge — лише в Git Bash
#### Rebase «переписує» історію для лінійного вигляду; merge чесно показує, що була окрема гілка.
}

::Конфлікти при merge::
Коли можуть виникнути конфлікти злиття згідно з лекцією? {
=Коли різні гілки змінили той самий фрагмент файлу по-різному і Git не може злити автоматично
~Лише коли на диску менше 1 ГБ вільного місця
~Тільки при роботі з одним файлом у репозиторії
#### Конфлікт — сигнал, що потрібне ручне рішення: хто «правий» у спірному рядку коду.
}

~Тільки під час написання README у Notepad
#### Гілки масштабують безпечну паралельну роботу — без них командна розробка швидко перетворюється на хаос.
}
