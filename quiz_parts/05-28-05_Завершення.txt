Що інструктор рекомендує зробити з усіма методами роботи з GitHub? {
  =Спробувати кожен, відпрацювати й зрозуміти його особливості
  ~Використовувати лише один «найпростіший» і забути решту
  ~Чекати наступного курсу, не практикуючи самостійно
  #### У фіналі наголошено: CLI, Desktop, clone, push/pull — усе варто пройти руками, а не лише подивитися демо.

  ~Перейменувати myProject на test і більше не чіпати
  #### Інструктор нагадує: myProject так і не було відправлено на GitHub — це практична вправа закріпити push і зв'язок local↔remote.

  ~Вона замінює всі наступні курси з Git
  ~Без неї GitHub закриває безкоштовний акаунт
  #### У транскрипті сказано: вправа може торкатися conflict/rebase — це свідоме закріплення, а не формальність.

::Завершення — інструменти курсу::
Які способи роботи з віддаленим кодом пройшли в блоці курсу? {
  =Командний рядок (push/pull/clone), GitHub Desktop, архів, fork і клонування
  ~Лише вебредактор GitHub без локальної копії
  ~Лише електронна пошта для обміну файлами
  #### У завершальному блоці узагальнюються всі шляхи отримати й синхронізувати код — від zip до повноцінного clone.
}

::Завершення — локальний і віддалений зв'язок::

  =Зв'язок локальної системи з GitHub (наприклад, через SSH або HTTPS)
  ~Обов'язковий платний план GitHub Team
  ~Видалення всіх локальних комітів
  #### Без налаштованого з'єднання push неможливий — це логічне продовження уроків про SSH і remote origin.
}

::Завершення — конфлікти в практиці::

  ~Без rebase Git не зберігає файли
  ~Конфлікти виникають лише в навчальних відео
  #### Навіть проста передача myProject на GitHub може зіткнути вас із відставанням гілок — це нормальний наступний крок після базового push.
}

  #### «Спробуйте самостійно» — ключова фраза завершення: інструктор дає карту, але м'язова пам'ять з'являється у вашій власній практиці.
