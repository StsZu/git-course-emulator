::Імперативна форма повідомлення::
Який стиль повідомлень комітів рекомендується в курсі як загальноприйнята практика? {
=Наказова (імперативна) форма: «Add file», «Fix bug», а не «Added file» чи «I fixed bug»
~Опис у минулому часі від першої особи: «Я виправив помилку вчора»
~Випадковий набір символів без змістового навантаження
#### Імператив читається як інструкція до коду: «Цей коміт зробить X» — так легше читати історію проекту.
}

::Один коміт — одна зміна::
Який принцип формування комітів підкреслює лектор у цьому відео? {
=Один коміт = одна логічна зміна, функція або оновлення, яке можна відновити окремо
~Усі зміни за день обов'язково збираються в один коміт о 18:00
~Коміт створюється лише раз на тиждень незалежно від обсягу роботи
#### Атомарні коміти спрощують відкат, code review і розуміння історії: кожна версія — завершений крок.
}

::git commit --amend --no-edit::
Навіщо в прикладі використовують git commit --amend --no-edit? {
=Щоб доповнити попередній коміт забутими змінами без нового повідомлення та без зайвого запису в історії
~Щоб видалити останній коміт і втратити всі його зміни назавжди
~Щоб автоматично створити нову гілку з виправленим іменем
#### Amend дозволяє «дописати» до щойно зробленого коміту, якщо ви забули рядок — без шуму зайвих записів у логу.
}

::git diff без --cached::
Чому після git add звичайний git diff може нічого не показати? {
=Бо зміни вже в staging area, а звичайний diff порівнює робочу директорію зі staging/останнім комітом
~Бо Git видаляє diff після першого коміту в репозиторії
~Бо diff працює лише з бінарними файлами, не з текстовими
#### Після add файл «піднято» з working tree в index — стандартний diff більше не бачить різниці там, де ви очікуєте.
}

::git diff --cached::
Яку задачу вирішує git diff --cached (або --staged)? {
=Показує різницю між staging area і останнім комітом — що саме потрапить у наступний commit
~Порівнює дві довільні гілки на віддаленому сервері
~Відновлює видалені файли з кешу браузера
#### --cached відповідає на питання: «Що я вже підготував до коміту, але ще не зберіг?»
}

::git restore --staged::
Що робить git restore --staged <file> у сценарії з відео? {
=Повертає файл із staging area назад у робочу директорію, залишаючи локальні правки
~Видаляє файл з диска і з усієї історії комітів безповоротно
~Автоматично публікує файл на GitHub
#### Unstage — скасувати «підготовку до коміту», не втрачаючи ваших редагувань у файлі.
}

::Заголовок і тіло коміту::
Як у відео формують розгорнуте повідомлення коміту в інтерактивному режимі? {
=Перший рядок — заголовок, після порожнього рядка — розширений опис (тіло) змін
~Усе повідомлення обов'язково пишеться одним словом без пробілів
~Повідомлення коміту завжди генерується Git автоматично з імен файлів
#### Структура title + body дає короткий огляд у git log і деталі для тих, хто розбирає історію глибше.
}

::Найважливіший шлях Git::
Який робочий цикл лектор називає «найважливішим шляхом» у взаємодії з Git? {
=Редагування файлів → staging area (git add) → збереження версії (git commit)
~Клонування → push → видалення локальної папки .git
~Створення гілки → merge → rebase у довільному порядку без комітів
#### Цей цикл — фундамент: усе інше (гілки, відновлення, GitHub) спирається на вміння правильно комітити.
}

::Untracked новий файл::
Який статус має щойно скопійований файл test1.txt до git add? {
=Untracked — Git бачить файл, але ще не відстежує його зміни
~Modified — файл уже вважається зміненим відносно останнього коміту
~Committed — файл автоматично входить до попереднього коміту
#### Новий файл спочатку «чужий» для Git; add переводить його під контроль версій.
}

::Modified після редагування::
Що показує git status після зміни вже відстежуваного файлу? {
=Статус modified — файл змінено в робочій зоні і потребує add/commit
~Статус untracked, ніби файл з'явився вперше
~Порожній вивід, бо Git не відстежує повторні редагування
#### Відстежуваний файл при редагуванні стає modified — це сигнал пройти цикл add → commit знову.
}
