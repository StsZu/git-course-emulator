::Визначення Pull Request::
Що таке Pull Request у формулюванні уроку? {
  =Процедура ілюстрації, перегляду та погодження інтеграції запропонованого коду в цільовий репозиторій
  ~Автоматичне клонування репозиторію на локальний диск
  ~Команда git pull у терміналі без участі людей
  #### PR — соціальний і технічний механізм: ви пропонуєте зміни, інші рев'юять і вирішують, чи merge.
}

::Коли створювати PR::
У якому сценарії урок демонструє створення Pull Request? {
  =Після змін у fork чужого репозиторію, коли хочете запропонувати їх автору
  ~Одразу після git init у новій папці
  ~Після видалення всіх гілок у локальному репозиторії
  #### Типовий open-source шлях: fork → зміни → PR до upstream-репозиторію.
}

::Порівняння гілок::
Що GitHub показує під час створення Pull Request? {
  =Порівняння гілок і змін у файлах між вашою копією та цільовою гілкою автора
  ~Лише список усіх користувачів GitHub
  ~Автоматично згенерований README без diff
  #### PR будується на diff: видно, що саме ви пропонуєте додати, змінити або видалити.
}

::Able to merge::
Що означає статус «able to merge» при створенні PR у демонстрації? {
  =Злиття можливе без конфліктів на поточний момент
  ~Автор уже автоматично прийняв ваші зміни
  ~Ваш fork видалено з платформи
  #### Цей статус підказує, що технічно зміни можна об'єднати; рішення все одно за автором або рев'юерами.
}

::Зелені рядки в diff::
Що позначають зелені рядки в перегляді змін Pull Request? {
  =Нові рядки, які ви пропонуєте додати до файлу
  ~Видалений код, який автор має відхилити
  ~Службові рядки GitHub, які не потраплять у код
  #### У diff зелений колір — додавання; саме так рев'юер бачить ваш внесок.
}

::Цільова гілка PR::
У куди автор демонстрації пропонує об'єднати свої зміни? {
  =У гілку main (master) оригінального репозиторію автора
  ~У приватний репозиторій без назви
  ~У локальну папку .git на комп'ютері автора
  #### PR зазвичай націлений на основну гілку upstream-репозиторію після перегляду.
}

::Роль автора PR::
Чи можна втрутитися в чужу гілку через PR без відома власника? {
  =Ні, без участі автора процес неможливий — він приймає, коментує або відхиляє пропозицію
  ~Так, merge відбувається автоматично через 24 години
  ~Так, якщо репозиторій публічний
  #### PR — запрошення до діалогу, а не самовільне вторгнення в чужий код.
}
